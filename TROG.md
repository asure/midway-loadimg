## Category 1: `+META` Derived Images

### The Problem

Some `+META` entries reference a base name that does **not exist** in any IMG
file. Example from `TROG.LOD` line 294:

```
----> BLOOVHAND1+VHAND1L,BLOOVHAND2+VHAND2L
```

- `BLOOVHAND1+VHAND1L` â†’ base name `BLOOVHAND1` â†’ found in `TROGPLAY.IMG`
- `BLOOVHAND2+VHAND2L` â†’ base name `BLOOVHAND2` â†’ **NOT found in any IMG**

The reference output gives `BLOOVHAND2` a SAG with 320 bytes of pixel data.
LOAD.EXE must be **generating derived pixel data** from another source,
likely by transforming `BLOOVHAND1`'s data using the `+VHAND2L` modifier.

### What `+META` modifiers might mean

Looking at the suffix patterns in Trog:

| Suffix | Pattern | Possible Meaning |
|--------|---------|-----------------|
| `+STP0L` | `*STP0+STP0L`| "Stop frame 0" â€” share same animation frame |
| `+GHL1L` | `GWEN*+GHL1L` | "Gwen head" â€” character-specific overlay |
| `+VHAND1L` | `*VHAND1+VHAND1L` | "Vertical hand" â€” hand position variant |
| `+VHAND2L` | `*VHAND2+VHAND2L` | "Vertical hand 2" â€” different hand position |
| `+HHAND2L` | `*HHAND2+HHAND2L` | "Horizontal hand 2" |
| `+eggL` | `*egg+eggL` | Egg/object variant |

The pattern `BLOOVHAND2+VHAND2L` suggests:
- Base name `BLOOVHAND2` should be a variant of `BLOOVHAND1`
- The `+VHAND2L` modifier indicates "apply vertical hand position 2"
- Pixel data is probably generated by **translating** `BLOOVHAND1`'s pixels
  by the offset encoded in the suffix

### What LOAD.EXE likely does

1. Look up base name in IMG â†’ if found, use its pixel data directly
2. If NOT found, treat it as a **derived image**:
   - Find the nearest matching base image (strip trailing number/letter)
   - Apply the suffix modifier (offset, flip, palette swap)
   - Generate pixel data and encode it

### Impact on SAG Cascade

When `BLOOVHAND2` is skipped (base not found), it gets:
- No `ImageEntry` (g.n_images not incremented)
- No IRW data
- No TBL entry

The reference has `BLOOVHAND2` at IMGTBL.ASM SAG[62] with 320 bytes of
properly encoded data. Our tool skips it. This:
1. Leaves a 320-byte gap in the IRW (our data is 320 bytes shorter)
2. Shifts all subsequent SAGs by 320 bytes
3. All later images in IMGTBL.ASM have wrong SAGs

### How the cascade propagates

```
IMGTBL.ASM: BLOOVHAND2 missing â†’ -320B
  â†“
TROGDAT.TBL (section 2): inherits -320B â†’ more diffs â†’ -235168B at SAG[401]
  â†“
TROGENEM.TBL: starts after cascade â†’ -1764B at SAG[0] (was -320B + intermediate)
  â†“
TROGSPRG.TBL: same -1764B offset (cancelled by BBB differences? â†’ ALL MATCH)
  â†“
TROGWHL.TBL: same -1764B offset (â†’ ALL MATCH)
```

### Possible Fixes

**Option A: Skip +META entries with missing base names (current behavior)**
- Pro: Simple, no crash
- Con: Missing images cascade to all subsequent SAGs

**Option B: Generate placeholder data for missing base names**
- Encode a minimal 1Ă—1 pixel for the missing entry
- SAG values would be ~1 byte off instead of 320 bytes
- Still not perfect, but cascade is smaller

**Option C: Derive pixel data from the nearest matching image**
- If `BLOOVHAND2` not found, try `BLOOVHAND1`
- Encode `BLOOVHAND1`'s data under `BLOOVHAND2`'s label
- This is what LOAD.EXE likely does
- Requires understanding the suffixâ†’transformation mapping

**Option D: Analyze LOAD.EXE's derived image algorithm**
- Requires the original LOAD.EXE binary or extensive reverse engineering
- Would need to map each suffix (+VHAND2L, etc.) to a specific transformation
- Not feasible without more documentation

## Category 2: Motorola ASM Format

### The Problem

Our tool outputs Intel-style hex (`03000000H`) while the ref uses
Motorola-style (`>3000000`). Additionally, the ref separates SAG and PAL
onto separate `.long` lines while our tool combines them:

```asm
; Reference (Motorola, separate lines):
.long	>3005A00
.long	PINK16

; Our output (Intel, combined line):
.long   03005a00H,PINK16
```

### What needs to change

The Motorola format is required by the TMS34010 GSP assembler (`gspa`):

| Feature | Motorola (needed) | Intel (current) |
|---------|-------------------|-----------------|
| Hex prefix | `>3000000` | `03000000H` |
| Long syntax | `.long` | `.long` |
| SAG/PAL | Separate lines | Combined on one line |
| File headers | Has `.FILE`, `.OPTION`, `.include`, `.even` | Just `.DATA` |

### Implementation plan

In `/OLD` mode, `write_image_tbl()` should:
1. Use `>%X` format instead of `0%XH` for hex values
2. Output SAG and PAL on SEPARATE `.long` lines
3. The caller (or a file-level wrapper) should emit the file headers

## Rule of Thumb for triage

When the reference output shows a SAG for a label whose base name doesn't
exist in any `.IMG` file, that label is a **derived image**. Skip it for
now and note it in the cascade tracking below.

## Cascade Tracking

```python
# To check cascade magnitude for each file:
python3 << 'EOF'
import re
def get_base(lines):
    pairs, label = [], None
    for l in lines:
        if l.strip().endswith(':'): label = l.strip()[:-1]
        m = re.search(r'(?:>|0?)([0-9a-fA-F]+)H?(?:,|$)', l)
        if m and '.long' in l and label:
            val = int(m.group(1), 16)
            if val > 0x1000000: pairs.append((label, val))
            label = None
    return pairs

for f in ['IMGTBL.ASM', 'TROGDDAT.TBL', 'TROGENEM.TBL', ...]:
    our = get_base(open(f).read().splitlines())
    ref = get_base(open(f'../reftrog/{f}').read().splitlines())
    first = next((i for i,(o,r) in enumerate(zip(our,ref)) if o[1]!=r[1]), None)
    if first is not None:
        print(f'{f}: first diff at SAG[{first}] {our[first][0]}: '
              f'our=0x{our[first][1]:X} ref=0x{ref[first][1]:X}')
EOF
```
